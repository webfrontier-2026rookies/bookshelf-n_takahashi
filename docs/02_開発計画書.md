# 開発計画書 — 書籍レビュー共有アプリ「BookShelf」

---

## 1. この計画書の使い方

要件定義書が「何を作るか」なら、本書は **「いつ・どの順番で・どうやって作るか」** です。
計画は守るためではなく、**ズレを早く検知するため**にあります。遅れたら計画を責めるのではなく、計画を更新してください。毎日終わりに「予定と実績」を5分で振り返ることを推奨します。

---

## 2. 全体戦略：垂直スライスで進める

レイヤーごと（先に全部のDB → 次に全部のBFF…）に作ると、最後まで何も動かず、統合で破綻します。
代わりに **「1機能を、フロント→BFF→バックエンドまで縦に貫いて動かす」垂直スライス** で進めます。

```
✗ 水平: [全DB] → [全gRPC] → [全GraphQL] → [全画面]   (最後まで動かない)
✓ 垂直: [認証を端から端まで] → [書籍登録を端から端まで] → ...  (毎日動くものが増える)
```

最初の1スライス（認証）が全層を貫通した時点で、**アーキテクチャの「配線」はほぼ完成**します。以降は同じ型の繰り返しなので加速します。

## 4. 日次計画（5日 + 予備2日）

> 時間は目安です。詰まったら**第6章のタイムボックス**ルールを使ってください。

### Day 1 — 環境構築とアーキテクチャ理解（M1）

- [✅] 雛形リポジトリを clone し、READMEに従って全app起動
- [✅] 4つのapp（web / bff / user-service / catalog-service）の役割を図に書いて理解する
- [✅] Prismaスキーマ（User/Book/Review）を確認・作成し、マイグレーション実行
- [✅] **疎通確認**: ダミーのgRPC 1本（例: ping）をbff→serviceで通す
- [✅] GitHub: リポジトリ準備、`main`保護、最初のPRを出してフロー確認

> Day1のゴールは「動くものを作る」ではなく「全部が起動して繋がる」こと。ここを焦らない。

### Day 2 — 認証スライス（M2）

- [✅] user-service: SignUp / Login の command 実装（パスワードハッシュ化、ULID採番）
- [✅] user-service: JWT発行
- [✅] bff: login/signup mutation、JWT検証ガード、認証コンテキスト（CLS or NestJSのcontext）
- [✅] web: ログイン／サインアップ画面（React Hook Form + Zod）、Zustandで認証状態保持
- [✅] web: URQL + Codegen のセットアップ（型生成が回ること）
- [✅] PR → レビュー → マージ

### Day 3 — 書籍CRUDスライス（M3）

- [ ] catalog-service: RegisterBook (command) / ListBooks・GetBook (query)
- [ ] bff: 対応する query / mutation、認証が要る操作にガード
- [ ] web: 書籍一覧（検索・ページネーション）、書籍登録フォーム、書籍詳細（枠だけ）
- [ ] Storybook: 書籍カード等のコンポーネントを1〜2個カタログ化
- [ ] PR → レビュー → マージ

### Day 4 — レビュー機能とN+1対策（M4）★最重要日

- [ ] catalog-service: PostReview (command)、レビュー一覧 query、`ListReviewsByBookIds` / `GetUsersByIds` バッチAPI
- [ ] bff: 書籍詳細でレビュー＋投稿者を返す。**まず素朴に実装してN+1を観測**（ログでgRPC呼び出し回数を見る）
- [ ] bff: **DataLoaderを導入してN+1を解消**。前後の呼び出し回数を記録（設計メモ用）
- [ ] web: レビュー投稿フォーム、書籍詳細にレビュー一覧表示
- [ ] commands側のユニットテスト（Jest）を書く
- [ ] PR → レビュー → マージ

> この日が課題の核心。**「N+1を自分の目で観測してから直す」** を必ず体験すること。最初から完璧に書こうとしない。

### Day 5 — 認可・マイページ・仕上げ（M5）

- [ ] レビュー編集・削除（command）。**サーバ側で所有者チェック（認可）**
- [ ] web: 他人のレビューには編集・削除ボタンを出さない／自分のは出す
- [ ] マイページ（自分のレビュー一覧）
- [ ] E2E（Playwright）でハッピーパス1本（ログイン→書籍登録→レビュー投稿）
- [ ] `docs/design.md` を書く（設計判断・N+1・認証認可・CQRS）
- [ ] PR → レビュー → マージ

### Day 6–7 — 予備・振り返り

- [ ] 積み残し対応、リファクタ、エラーハンドリング改善
- [ ] README整備、振り返りメモ作成
- [ ] メンターとの最終レビュー準備（説明できるよう整理）

## 5. GitHubフローの実践

```
main (常にデプロイ可能・壊さない)
  └─ feat/auth          ← 機能ごとにブランチを切る
       └─ commit, commit ...
       └─ Pull Request → メンターがレビュー → main へマージ
```

- **1機能 = 1ブランチ = 1 PR**。PRが大きくなりすぎたら分割する（レビューできない巨大PRは実務で嫌われる）。
- ブランチ名例: `feat/auth`, `feat/book-register`, `feat/review-dataloader`, `fix/login-validation`
- コミットメッセージ: 1行目に要約、本文に「なぜ」。例:

  ```
  feat: add DataLoader for review authors

  書籍詳細でレビュー投稿者を取得する際にN+1が発生していたため、
  ユーザーIDをバッチ解決するDataLoaderを導入した。
  gRPC呼び出しが N+1 回 → 2 回に削減。
  ```

- PR説明には「何を・なぜ・どう確認したか」を書く。**レビュー依頼は早めに、小さく**。

## 6. 詰まったときのルール（タイムボックス）

初級者が1週間で最も陥りやすいのは「1つの問題に半日溶かす」ことです。次を厳守してください。

1. **30分ルール**: 同じエラーで30分進まなければ、いったん手を止める
2. 切り分け: 「どの層で・何を期待し・実際に何が起きたか」を1メモにする
3. 最小再現: 問題を最小のコードで再現してみる
4. **45分でメンターに質問**: 上記メモを添えて相談する（質問は恥ではなく、評価対象）
5. 迂回も選択肢: 詰まった機能は後回しにし、別スライスを進めて全体を止めない

## 7. スコープ調整（間に合わない場合の優先順位）

時間が足りないときは、**下から削ります**。上は死守してください。

| 優先度       | 機能                                                     | 方針           |
| ------------ | -------------------------------------------------------- | -------------- |
| 必須（死守） | 認証・書籍一覧/詳細・レビュー投稿・**N+1対策**・**認可** | 学習目的の核心 |
| 重要         | 書籍検索・ページネーション・マイページ                   | 余裕があれば   |
| 任意         | E2E複数ケース・Storybook網羅・凝ったUI                   | 最後           |

> 「全機能未完だがN+1と認可を深く理解した」状態は、「全機能できたが仕組みを説明できない」状態より**高く評価されます**（要件定義書 第14章）。

## 8. リスクと対策

| リスク                 | 兆候                           | 対策                                        |
| ---------------------- | ------------------------------ | ------------------------------------------- |
| 環境構築でDay1を超過   | Day2朝に疎通できていない       | 早朝にメンターへSOS。雛形の前提を疑う       |
| gRPC/proto配線でハマる | サービス間で型・接続エラー連発 | protoは小さく1本通してから増やす            |
| Codegenが回らない      | 型が生成されない               | スキーマURL・スクリプト設定をメンターと確認 |
| N+1対策で行き詰まる    | DataLoaderの使い方が不明       | まず素朴実装で動かし、観測してから最適化    |
| 完璧主義で進まない     | 1画面のUIに半日                | 機能優先。UIは「動けば及第点」で次へ        |
